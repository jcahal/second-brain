🐳 EC2 + Docker Compose on AWS Free Tier

Manual walkthrough — Ubuntu 22.04 · t2.micro · No Elastic Beanstalk

Prerequisites

  • AWS account with free tier active
  • A working docker-compose.yml in your project
  • SSH client (built into macOS/Linux; use PuTTY or Windows Terminal on Windows)
  • Your project code in a Git repo (or ready to SCP up)
1
Launch a t2.micro Instance
AWS Console → EC2 → Launch Instance

Go to EC2 → Instances → Launch Instance. Fill in the following:

Click Launch Instance. Wait ~60 seconds for the status to reach 2/2 checks passed.

2
SSH Into the Instance
From your local machine terminal

Get the Public IPv4 DNS from the EC2 instance detail panel, then:

Fix .pem permissions (macOS/Linux only — required) 
chmod 400 ~/Downloads/my-key.pem
Connect 
ssh -i ~/Downloads/my-key.pem ubuntu@ec2-xx-xx-xx-xx.compute-1.amazonaws.com
⚠️ The default user is ubuntu for Ubuntu AMIs. Not ec2-user, not root.

If you see a shell prompt like ubuntu@ip-172-xx-xx-xx:~$ — you're in.

🔥 Most common failure here: SSH port 22 not open in your security group, or wrong source IP. Go back to EC2 → Security Groups and verify.
3
Install Docker + Compose Plugin
Run these inside your SSH session

Use Docker's official install script — it handles Ubuntu correctly and installs the v2 Compose plugin:

# Update apt and install prerequisites
sudo apt-get update
sudo apt-get install -y ca-certificates curl

# Add Docker's GPG key and repo
sudo install -m 0755 -d /etc/apt/keyrings
sudo curl -fsSL https://download.docker.com/linux/ubuntu/gpg -o /etc/apt/keyrings/docker.asc
echo "deb [arch=$(dpkg --print-architecture) signed-by=/etc/apt/keyrings/docker.asc] https://download.docker.com/linux/ubuntu $(. /etc/os-release && echo $VERSION_CODENAME) stable" | sudo tee /etc/apt/sources.list.d/docker.list

# Install Docker Engine + Compose plugin
sudo apt-get update
sudo apt-get install -y docker-ce docker-ce-cli containerd.io docker-compose-plugin

# Let ubuntu user run docker without sudo
sudo usermod -aG docker ubuntu
newgrp docker
Verify it worked 
docker --version
docker compose version   # note: no hyphen (v2 plugin)
💡 docker compose (space, v2 plugin) vs docker-compose (hyphen, v1 standalone). The plugin is what you want — it's what gets installed above.
4
Get Your Project onto the Instance
Two options — pick one
Option A — Clone from Git (recommended)
# On the EC2 instance
git clone https://github.com/yourname/your-repo.git
cd your-repo
Option B — SCP from your local machine
# Run this on your LOCAL machine, not on EC2
scp -i ~/Downloads/my-key.pem -r ./your-project ubuntu@ec2-xx-xx-xx-xx.compute-1.amazonaws.com:~/
💡 Git is cleaner for iteration. You can git pull to deploy updates instead of reuploading files.
5
Handle Environment Variables
Don't commit secrets — create a .env file on the instance

If your Compose file uses a .env file, create it directly on the instance. Don't put secrets in your git repo.

# Create a .env in your project directory on EC2
nano .env

# Paste in your variables, e.g.:
DATABASE_URL=postgres://user:pass@db:5432/mydb
SECRET_KEY=your-secret-here
✅ Make sure .env is in your .gitignore. The file lives only on the server — never in version control.
6
Start Your Stack
Inside your project directory on EC2
# First run: build images and start in foreground to see logs
docker compose up --build

# Once it looks healthy, Ctrl+C and start detached
docker compose up -d --build
Useful commands while debugging 
docker compose ps               # status of all services
docker compose logs -f           # follow all logs
docker compose logs -f web       # follow logs for one service
docker compose exec web bash     # shell inside a running container

Hit your EC2 public IP in a browser. If it doesn't load, check your security group ports first.

7
Survive Reboots
Make your stack start automatically when the instance restarts

The quick way: add restart: always to each service in your Compose file, then enable Docker to start on boot:

# In docker-compose.yml, each service gets:
services:
  web:
    image: ...
    restart: always   # ← add this line
# On the EC2 instance — enable Docker daemon on boot
sudo systemctl enable docker
💡 restart: always tells Docker to restart the container if it crashes or if the Docker daemon starts. Combined with systemctl enable docker, your stack comes back up after any reboot automatically.
Test it
sudo reboot
# Wait ~30s, then SSH back in and run:
docker compose ps
8
Deploying Updates
After making code changes
# SSH into your instance, go to project dir, then:
git pull
docker compose up -d --build

# Old containers are replaced; volumes (your data) are preserved
✅ Named volumes persist across docker compose up --build. Your database data survives redeployments unless you explicitly run docker compose down -v.
✓
Launch Checklist
Things to verify before calling it done
  • □ Security group has ports 22 (SSH), 80 (HTTP), and your app port open
  • □ docker compose ps shows all services as Up
  • □ App accessible via EC2 public IP in browser
  • □ restart: always set on all services
  • □ sudo systemctl enable docker run
  • □ .env is in .gitignore
  • □ Rebooted and confirmed stack came back up automatically
